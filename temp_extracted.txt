1. Project Overview
ISAES is a full-stack educational advisory platform that combines a rule-based expert system with an AI-powered chatbot and a formal report generation pipeline. The system ingests student academic data and applies a structured knowledge base of inference rules to produce personalised academic recommendations, risk assessments, and printable advisory reports.

3.3 Expert System — Inference Engine
The inference engine is the intellectual core of the system. Built using Experta (a Python port of CLIPS), it supports both forward and backward chaining over a YAML-defined knowledge base.
Inference Modes
Forward chaining — given a student's data, fire all applicable rules automatically and derive conclusions (e.g., GPA < 2.0 AND attendance < 60% → HIGH RISK flag)
Used for: risk assessment, scholarship alerts, attendance warnings, progression checks
Backward chaining — given a goal, work backward through the rule chain to verify conditions (e.g., Is this student eligible for the Honours track?)
Used for: eligibility verification, track qualification, award criteria
Rule Categories
Category
Example Rule
Risk Assessment
If CGPA < 2.5 and failed_subjects >= 2, then Risk = HIGH
Subject Recommendation
If interest = 'Data Science' and math_grade >= B, then recommend ML electives
Remediation Advice
If subject = 'Statistics' and grade = D, then recommend tutoring + resource links
Track Eligibility
If CGPA >= 3.5 and no backlogs, then eligible for Honours / Research track
Scholarship Alert
If CGPA >= 3.7 and income_band = LOW, then flag for scholarship application
Attendance Warning
If attendance < 75% in any subject, then trigger attendance advisory
Progression Check
If year = 2 and credit_completion < 70%, then risk of year drop
Rule File Format (YAML)
Sample Rule — R001: high_risk_cgpa
rule_id: R001
name: high_risk_cgpa
description: Flag student as high risk if CGPA is critically low
conditions:
  - field: cgpa
    operator: lt
    value: 2.0
  - field: failed_subjects
    operator: gte
    value: 2
conclusion:
  risk_level: HIGH
  recommendation: "Immediate academic counselling required."
  action_tags: [counselling, remediation, parent_alert]
priority: 1
3.4 Chatbot — AI Advisory Assistant
The chatbot provides a natural-language interface over the expert system's structured output. It is powered by the Claude API and grounded in each student's academic context.
Architecture
FastAPI WebSocket endpoint receives user message
System prompt includes: student profile data + current inference recommendations + conversation history
Claude API (claude-sonnet) generates a contextually grounded response
Response is streamed back to the frontend token-by-token
Chatbot Capabilities
Answer questions about grades, CGPA, and attendance in plain language
Explain advisory recommendations in student-friendly terms
Suggest study resources, career paths, and elective choices
Alert about deadlines, backlogs, and scholarship windows
Escalate serious concerns (dropout risk, mental health flags) to a human advisor
Session Management
Redis stores conversation history keyed by session_id. Each session is scoped to one student and expires after 24 hours of inactivity. Session history is passed to Claude on each turn to maintain context.
3.5 Report Generator
Each advisory report is a formal, printable document. Reports are generated server-side as PDFs using WeasyPrint (HTML-to-PDF) and stored linked to the student record. Every run produces a new versioned report; old versions are archived.

9. Key Design Decisions
Why Experta for the Expert System?
Experta is a production-ready Python port of CLIPS with native Rete algorithm-based forward chaining. Rules are human-readable YAML files editable by non-developers, version-controllable in Git, and hot-reloadable without a server restart.
Why FastAPI over Django?
FastAPI is async-native and auto-generates OpenAPI/Swagger documentation. Pydantic validation is a natural fit for rule schemas and student data. It is lighter weight than Django for an API-only backend and has excellent performance under concurrent load.
Why Claude for the Chatbot (not a custom model)?
The system already has a structured inference engine for rule-based reasoning. Claude's role is purely language — translating structured recommendations into natural conversational guidance. Using the API eliminates fine-tuning cost, model hosting overhead, and brings Anthropic's safety capabilities built in.
Why Server-side PDF Generation?
Consistent formatting is guaranteed regardless of the student's browser or OS. The advisory report is an official institutional document and must look identical every time. Server-side generation also allows reports to be stored, versioned, and audited centrally.
